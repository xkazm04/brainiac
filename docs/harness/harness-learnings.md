# brainiac — harness learnings

## Structural facts
- **2026-07-14** — The codebase is a Rust workspace of 8 crates (`brainiac-{core,fixtures,store,gateway,pipeline,publish,eval,server}`) + a Next.js 15/React 19 console. ~217 source files (78 Rust, 139 console TS/TSX), ~50k LOC.
- **2026-07-14** — Auth model: env tokens (`BRAINIAC_TOKENS` JSON) carry `scopes: None` = **unrestricted** (`AuthContext.allows` returns true for any scope); managed `brk_` tokens carry explicit scopes (`kb:read`/`kb:publish`/`read`/`write`/`admin`, where `admin` implies all). `auth_of(state, headers, scope)` passes the scope string straight to `allows()`. `crates/brainiac-server/src/auth.rs`.
- **2026-07-14** — RLS: every runtime query runs under the caller's `store.scoped_tx(&principal)`; the RLS-bypassing **admin pool** is built per-caller via `brainiac_store::admin_pool(url)` in `main.rs` and passed to sweeps. `AppState` (`http.rs:28`) holds **no** admin pool — org-true (cross-visibility) computation in a request handler currently has no pool to use without plumbing one in.
- **2026-07-14** — The document "anti-rot" invariant lives in `documents.dirty_at` + `mark_dirty_for_memory` (uses `COALESCE(dirty_at, now())`, so it can't advance an already-dirty timestamp) and `insert_revision` / `compose_tick` (which clear it). Two findings (R06 worker side, R10 store side) describe the SAME lost-update from both ends.
- **2026-07-14** — The console authenticates all users with **one shared passcode** and calls the backend with **one shared service token** (`BRAINIAC_API_TOKEN` via `configFromEnv()`). There is no per-user identity; the backend's per-team maintainer authz is enforced against that constant token. Documented as a deliberate single-operator posture (2026-07-14 decision).

## Conventions enforced
- **2026-07-14** — Mutating server endpoints gate on an explicit role (`is_maintainer` / `is_any_maintainer`), not on RLS visibility. `doc_approve` is the reference pattern; `doc_edit` was brought in line (fix `538b7a5`).
- **2026-07-14** — "Decide once" state transitions are self-guarding: precondition in the UPDATE `WHERE` + `FOR UPDATE` on the actionable read, `rows_affected == 0` → 409 (promotion approve/reject, `d69c8f3`).
- **2026-07-14** — Rust gate = `cargo check --workspace --all-targets` (baseline 0 errors). Console gates = `npx tsc --noEmit` + `npx vitest run` (50 unit tests, no DB). The `*_pg.rs` integration tests need a live Postgres + `DATABASE_URL` and are NOT run in the harness. Note: a running `brainiac.exe` locks `target/debug/brainiac.exe`, so `cargo test --no-run` fails to link — stop the server or rely on `cargo check`.

## Anti-patterns to avoid
- **2026-07-14** — Treating RLS "can-see" as authorization "can-mutate" (R04 doc_edit — a read-scoped token could rewrite published prose).
- **2026-07-14** — Check-then-act on a governance decision without a row lock or self-guarding WHERE (R01#2 promotion TOCTOU).
- **2026-07-14** — Revalidating a list only on the success path: a client that loses a concurrency race keeps showing phantom rows. Revalidate on the already-decided failure path too (C05/C06/C11).
- **2026-07-14** — Fetch/HTTP without a timeout is pervasive across both stacks (`reqwest::Client::new()` and console `fetch`): R13 (Critical, worker hang), R17 (Confluence holds a DB txn), C03/C04. Fix as one theme (Wave 4).
